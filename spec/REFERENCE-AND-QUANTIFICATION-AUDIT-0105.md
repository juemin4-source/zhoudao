# 周道指称、数量与上下文词审计 v0.10.5

> 汉语口语会依靠上下文完成指称，程序源码不能偷偷依靠上下文。
> 名称只标识对象，不通过「们」「这些」「该」「其」等词暗示数量或篇章指代。
> 数量由容器、量词和操作结构表达；指代必须由显式名称或具有唯一语法绑定的上下文词表达。

---

## 一、总则

**`叫什么，由名称负责；有几个、是哪批、当前指谁，由语法负责。`**

自然汉语 → 展开隐含指称 → 删除依赖语境的省略、代词和模糊指称 → 周道源码

```
自然内语：                这些项目一个个处理
展开隐含指称：             从项目列表中逐项取出并处理每个项目
周道源码：                从项目列表中，每取一项记作项目，就处理（项目）。
```

---

## 二、复数与集合指称

### 2.1 `们` — 不作为复数语法 ❌

**裁决：** 周道不使用 `们` 给名称标复数，也不根据 `们` 判断一个值是不是集合。

**原因：**
1. 汉语名词不强制区分单复数（`一个项目`、`三个项目`、`处理项目`）
2. `们` 自然地附着于人或拟人对象（`朋友们`、`同学们`），`文件们`、`数字们` 带有拟人或童话语气
3. `用户们` 更像称呼一群人的称呼语，不是中性的程序数据名称

**不推荐：**
```
项目们
文件们
数据们
结果们
用户们
```

**推荐：**
```
项目列表
参与用户
待处理文件
结果序列
处理队列
来源
```

**原则：** 名称说明角色，语法说明数量。

---

### 2.2 `这些` / `那些` — 仅限解释层 ❌

**裁决：** `这些` `那些` 不进入规范可执行语法。

**原因：** 指示词天然要求前提（"哪些项目？"），编译器不能靠感觉回答上下文指代。

**可用场景：** 注释、教程、LSP 解释、AI 对程序的自然复述、错误信息。

```
源码：   从项目列表中，每取一项记作项目，就处理（项目）。
解释层： 从这些项目中逐项取出并处理。
```

规范的源码与解释语言形成两层：
- **规范源码：** 显式名称、唯一指称
- **解释语言：** 这些、它们、前面的项目

## 三、代词与篇章指称 ❌

### 3.1 `其`

**裁决：** 不进入普通表达式。

```
不推荐：    取得用户，其姓名为……
推荐：      用户的姓名
```

`其` 可能指用户，也可能指上一项结果。优先重复明确名称。

### 3.2 `它` / `它们` / `他们`

**裁决：** 不进入普通表达式。

```
不推荐：    读取文件，然后处理它。
推荐：      读取文件（路径），记作文件内容。处理（文件内容）。
```

### 3.3 `该` / `此` / `上述` / `前者` / `后者`

**裁决：** 不进入可执行语法。

```
不推荐：    对该项目进行处理
推荐：      处理（项目）
```

适合文档，不适合可执行源码。

---

## 四、有限制地允许的上下文词 ✅

### 4.1 正式上下文绑定词（允许）

| 上下文词 | 唯一绑定来源 | 示例 |
|---------|------------|------|
| `自己` | 当前类别方法的实例 | `自己的状态` |
| `错误` | 当前异常分支捕获的异常对象 | `错误` |
| `错误内容` | 当前异常的 message | `错误内容` |
| `当前错误` | 当前正在处理的异常 | `原样报出当前错误` |
| `本定义外层` | 当前定义的直接词法外层 | `指本定义外层的数量` |
| `下文所用……均指……` | 声明后的当前作用域 | `下文所用数量，均指全局的数量` |

### 4.2 `本` — 有限制地允许

不可自由使用：
```
本对象     ← 模糊，禁止
本项目     ← 模糊，禁止
```

可作为固定语法的一部分：
```
本定义外层     ← 由词法作用域机械确定 ✅
本模块         ← 由模块系统确定 ✅
```

### 4.3 `当前` — 有限制地允许

不可自由使用：
```
当前项目     ← 除非语言已经定义"当前项目"，否则禁止
```

可作为正式上下文概念：
```
当前错误     ← 只在异常处理区中有唯一绑定 ✅
```

## 五、量词与数量表达

### 5.1 可进入形式语法 ✅

这些词能形成明确量化，但每个必须有精确语义定义：

| 量词 | 状态 | 示例 |
|------|------|------|
| `每` | ✅ 已定义 | `每取一项记作……` |
| `所有` | 🔮 候选 | 待定义精确语义 |
| `任一` | 🔮 候选 | 待定义精确语义 |
| `至少一个` | 🔮 候选 | 待定义精确语义 |

### 5.2 不可进入可执行语法 ❌

```
一些
若干
多个
不少
大部分
相关
适当
通常
必要时
等等
```

这些在自然语言中有用，在程序中没有确定边界，只能出现在注释、需求或 AI 指令中。

## 六、`每` 与 `各` 的区分

### `每` — 已用于遍历语法 ✅
```
从来源中，每取一项记作项目，就……
```

### `各` — 需要谨慎 🔮
`各项目` 常表示集合中的所有成员，但不一定说明执行顺序，也可能暗示分别对应。

当前遍历继续用 `每取一项`，比 `各取一项` 更清楚。

### `分别` — 候选，待正式定义 🔮
```
将甲、乙分别赋给丙、丁     ← 隐含位置配对和数量相等
```

没有正式语义之前，不能把 `分别` 当普通副词随处插入。

---

## 七、量词 `一个` / `一项` / `一条`

**裁决：** 量词属于语法，不属于名称。

```
每取一项记作项目
↑            ↑
"一项" 回答"取得多少"
"项目" 回答"把取得的东西叫什么"
```

名称本身不应包含量词：
```
不推荐：    一个项目  某个项目  这一项
推荐：      项目  条目  文件  用户
```

---

## 八、中文名词与容器类型

中文名词不天然说明容器类型。`项目` 可以指单一项目，也可以指项目列表。

**原则：** 编译器不因名字推断类型。但规范示例和命名建议应鼓励：

| 角色 | 推荐命名 |
|------|---------|
| 单项 | `项目`、`条目`、`用户` |
| 序列 | `项目列表`、`项目序列`、`待处理项目` |
| 集合 | `项目集合` |
| 映射 | `项目映射`、`项目索引` |
| 迭代来源 | `来源` |
| 结果 | `处理结果`、`结果列表` |

**这不是语法规则**——编译器不能因为名字叫 `项目列表` 就认定它是列表，也不能因为叫 `项目` 就认定是单个对象。这是命名风格规范。

---

## 九、等待下一步审计的词

以下候选词与 `们`/`这些` 属于同一类母语陷阱：

```
其 / 该 / 此 / 当前 / 本
每 / 各 / 所有 / 任一
一个 / 一项 / 一条
分别 / 依次 / 逐个 / 一起
有 / 含有 / 属于 / 是 / 为
空 / 没有 / 没有值
相同 / 相等 / 就是
```

尤其后三组语义容易混淆：

```
空列表              ← 空的列表
没有列表            ← 列表不存在
列表没有值          ← 列表绑定到了 None？

甲与乙相同          ← 值相等？结构相同？对象身份？
甲 = 乙             ← 值相等（数学意义）
甲就是乙            ← 对象身份（is）
```
