# 运算符边界审计：% 与 //

> 依据「符号不够直观时，只有在汉语表达更清楚、更紧凑、更易组合的情况下才翻译；
> 翻译会变成长句或技术黑话时，宁可保留符号并由工具解释」这条补充原则。
> 目的：在裁决 % 和 // 前，拿到完整的语义边界和程序语境证据。

---

## 一、`%`（取余 / 模）语义差分

### 1.1 边界值表

| Python 表达式 | 结果 | 语义说明 | 汉语「余数」直觉是否匹配 |
|--------------|------|---------|----------------------|
| `5 % 2` | `1` | 5 ÷ 2 = 2 余 1 | ✅ 完全匹配 |
| `5 % -2` | `-1` | 5 ÷ (-2) = -2 余 -1 | ❌ 通常余数不说负数 |
| `-5 % 2` | `1` | -5 ÷ 2 = -3 余 1 | ❌ 被除数负，余数正 |
| `-5 % -2` | `-1` | -5 ÷ (-2) = 2 余 -1 | ❌ 余数为负 |
| `5.5 % 2` | `1.5` | 浮点取余 | ⚠️ "余"一般不用于小数 |
| `5 % 0` | `ZeroDivisionError` | 除零 | — |
| `0 % 5` | `0` | 零被除 | ✅ |

### 1.2 关键发现

Python `%` 不是"余数"（remainder），而是**模运算**（modulo）。两者在负数时的差异：

```
数学余数（remainder）：    被除数 - 商 × 除数，其中商向零取整
Python 模（modulo）：      被除数 - 商 × 除数，其中商向下取整

例：-5 ÷ 2
  商向零取整 = -2 → 余数 = -5 - (-2×2) = -1
  商向下取整 = -3 → 模    = -5 - (-3×2) = 1
```

在四则运算的日常生活中，"余"几乎只出现在**正整数范围**。负数余数没有普遍的教学基础。

### 1.3 嵌入完整代码

#### 场景 A：奇偶判断（% 最常用场景，全正整数）

```
# 方案 1：保留 %
如果数量 % 2 = 0，就显示【偶数】，不然就显示【奇数】。

# 方案 2：对……取余
如果数量对2取余 = 0，就显示【偶数】，不然就显示【奇数】。

# 方案 3：当前汉语（余）
如果数余2等于0，就依次给出数。
```

**扫读对比：** 方案 1 和 2 在一个量级 —— `% 2` 和 `对2取余` 认知成本都很低。方案 2 的介词结构 `对……取余` 将 `2` 夹在中间，比 `余2` 更完整但多 3 字。

#### 场景 B：分组/分桶（正整数）

```
# 方案 1
设桶号 = 用户序号 % 桶总数。

# 方案 2
设桶号 = 用户序号对桶总数取余。

# 方案 3
设桶号为用户序号余桶总数。
```

**扫读对比：** 方案 1 `%` 一眼扫完。方案 2 的 `对桶总数取余` 将 `桶总数` 夹在 `对` 与 `取余` 之间，视觉跨度更大。方案 3 `余` 乍看像并列词。

#### 场景 C：取余数判循环

```
# 方案 1
如果行号 % 26 = 0，就插入分页符。
设余量 = 总长度 % 每页行数。

# 方案 2
如果行号对26取余 = 0，就插入分页符。
设余量 = 总长度对每页行数取余。
```

#### 场景 D：负数出现在实际程序中

```
# 周期性索引偏移（负数场景有实际用途）
设偏移 = -3。
设实际索引 = （当前位置 + 偏移）% 总长度。

# 汉语候选
设实际索引 = （当前位置 + 偏移）对总长度取余。
```

**观察：** `%` 在此处的作用更接近循环回绕，而不是"除法余数"。"取余"在这里仍可理解，但"对……取余"的 `对` 在 `=` 右边出现时，阅读节奏需要调整。

## 二、`//`（向下取整除法）语义差分

### 2.1 边界值表

| Python 表达式 | 结果 | 语义说明 | 汉语「整除」直觉是否匹配 |
|--------------|------|---------|----------------------|
| `5 // 2` | `2` | 5 ÷ 2 = 2.5，向下 = 2 | ✅ "5整除2得2" |
| `-5 // 2` | `-3` | -5 ÷ 2 = -2.5，向下 = -3 | ❌ 通常理解"整除"是 -2 |
| `5 // -2` | `-3` | 5 ÷ (-2) = -2.5，向下 = -3 | ❌ 同上 |
| `-5 // -2` | `2` | -5 ÷ (-2) = 2.5，向下 = 2 | ✅ 巧合一致 |
| `5.5 // 2` | `2.0` | 浮点向下取整 | ⚠️ "整除"不用于小数 |
| `5 // 0` | `ZeroDivisionError` | 除零 | — |

### 2.2 关键发现

`//` 的语义严格等于 `floor(a / b)`，不同语言对负数时行为不同：

| 语言 | `-5 ÷ 2` | 规则 |
|------|---------|------|
| Python | `-3` | floor（向下取整） |
| C/Java | `-2` | truncate（向零取整） |
| 日常数学 | `-2` 或 `-3` | 看上下文 |

汉语「整除」在数学教育中通常表达的是**关系**（`6 能被 3 整除`），而不是**运算**（`5 ÷ 2 = 2 余 1` 中的商）。“整除”作为运算是程序员行话。

### 2.3 嵌入完整代码

#### 场景 A：分页（最常用场景，全正整数）

```
# 方案 1：保留 //
设总页数 = 总条数 // 每页条数。
如果当前索引 // 每页条数 = 当前页，就……
设当前页 = （当前索引 + 偏移）// 每页条数。

# 方案 2：对……取整
设总页数 = 总条数对每页条数取整。
如果当前索引对每页条数取整 = 当前页，就……
设当前页 = （当前索引 + 偏移）对每页条数取整。

# 方案 3：向下取整
设总页数 = （总条数 / 每页条数）向下取整。

# 方案 4：当前汉语
设页数为总数整除每页数量。
```

**扫读对比：**
- 方案 1 `//` 最精简，一眼看过去 2 个字符。
- 方案 2 `对……取整` 在 `总条数对每页条数取整` 中结构完整，但视觉跨度是 `//` 的 5 倍。
- 方案 3 `（…）/ … 向下取整` 太重，不适合作为中缀。
- 方案 4 `整除` 在正整数范围尚可，负数时误导。

#### 场景 B：二分查找的中间索引

```
# 方案 1
设中位 = （左 + 右）// 2。

# 方案 2
设中位 = （左 + 右）对2取整。

# 方案 3
设中位 = （左 + 右）/ 2 向下取整。
```

**观察：** 方案 1 `// 2` 是程序员通用表达，即使不确切知道"向下取整"也大致理解意图。方案 2 `对2取整` 在括号外，`（…）对2取整` 中 `对` 的前后关系不如 `//` 直观。

#### 场景 C：时间计算（负数场景）

```
# 假设 timezone 偏移 -3
设本地时间 = UTC时间 + 偏移。            ← 没争议，用 +
设本地日期 = UTC天数 + 偏移 // 24。      ← 但这里用 //
```

```
# 方案 1
设本地日期 = UTC天数 + 偏移 // 24。

# 方案 2
设本地日期 = UTC天数 + 偏移对24取整。
```

**// 的无可替代性：** `//` 可以干净地嵌入混合表达式 `偏移 // 24`。`对24取整` 在 `偏移` 右侧，与 `+` 在一起时 `+ 偏移对24取整` 的 `对` 读起来像介词，视觉上容易连到前面的 `天数 +`。

## 三、组合性测试

运算符在复合表达式中的表现是关键指标：

### 3.1 `%` 组合

```
# 保留 %
设结果 = （甲 % 乙）+ 丙。
设结果 = 甲 % （乙 + 丙）。
设结果 = 甲 * 乙 % 丙。

# 对……取余
设结果 = （甲对乙取余）+ 丙。
设结果 = 甲对（乙 + 丙）取余。
设结果 = 甲 * 乙对丙取余。
```

**边界问题：** `甲对（乙 + 丙）取余` 中，`对` 和 `取余` 之间的括号跨度大时，读者需要保持配对状态直到 `取余` 出现。`%` 是中缀符号，左右边界由符号自身定义。

### 3.2 `//` 组合

```
# 保留 //
设结果 = （甲 // 乙）+ 丙。
设结果 = 甲 // （乙 + 丙）。
设结果 = 甲 + 乙 // 丙。

# 对……取整
设结果 = （甲对乙取整）+ 丙。
设结果 = 甲对（乙 + 丙）取整。
设结果 = 甲 + 乙对丙取整。
```

**观察：** `甲 + 乙对丙取整` 中，`+` 的优先级高于 `对……取整` 还是低于？这个介词结构没有数学先例。而 `甲 + 乙 // 丙` 的优先级继承自 Python，程序员已知。

## 四、对补充原则的验证

> 符号不够直观时，只有在汉语表达更清楚、更紧凑、更易组合的情况下才翻译；
> 翻译会变成长句或技术黑话时，宁可保留符号并由工具解释。

### 4.1 `%` 检验

| 维度 | 保留 `%` | 候选 `对…取余` |
|------|---------|--------------|
| 清楚度 | ⚠️ `%` 对非程序员是百分号 | ✅ 对程序员和非程序员都明确 |
| 紧凑度 | ✅ 1字符 | ⚠️ 3-6字（含操作数） |
| 组合性 | ✅ 中缀符号，优先级有数学直觉 | ⚠️ 介词结构包裹操作数，长组合时配对负担 |
| 负数匹配 | ⚠️ Python `%` 非余数 | ❌ 「余」会误导负数时真正语义 |
| 浮点匹配 | ❌ 浮点取余在数学中不存在 | ❌ 同左 |
| LSP 解释 | ✅ 只需一句话"Python 模运算" | — |

**结论：** `对…取余` 在正整数范围优于 `%`，但 4/6 维度持平或劣势。汉语在 `%` 这个符号上的增益是**局部**的（仅正整数常见场景），在全局（负数、浮点、组合）反而有损耗。

### 4.2 `//` 检验

| 维度 | 保留 `//` | 候选 `对…取整` | 候选 `整除` |
|------|----------|--------------|-----------|
| 清楚度 | ⚠️ 需要说明"向下取整" | ✅ 自解释 | ❌ 语义不符负数 |
| 紧凑度 | ✅ 2字符 | ⚠️ 3-6字 | ✅ 2字 |
| 组合性 | ✅ 中缀，继承 Python | ❌ 介词结构，组合弱 | ⚠️ 中缀尚可 |
| 负数匹配 | ✅ Python 语义完整 | ✅ 语义准确 | ❌ 数学「整除」≠ floor |
| 浮点匹配 | ✅ | ✅ | ❌ "整"排除浮点 |
| LSP 解释 | ✅ 一句话"向下取整" | — | — |

**结论：** `整除` 有语义问题（负数/浮点）。`对…取整` 语义准确但组合性弱。`//` 虽然需要一次解释，但解释成本低且收益持久。

### 4.3 综合验证

这条审计暴露了一个以前没想到的角度：**汉语介词结构 `对……取` 作为中缀运算符的适配性。**

`甲对乙取余` 本质上把二元运算符变成了「前缀 + 中置 + 后缀」的三段式：

```
  甲  对  乙  取  余
  ↑       ↑       ↑
操作数1  介词  操作数2  动词  名词
```

这在简单表达式（`甲对2取余`）中没问题，但在组合表达式（`（甲 + 乙）对（丙 * 丁）取余`）中，读者需要一直记住 `对` 开始的配对直到 `取余` 出现——括号内的表达式越长，这种配对负担越大。

而 `%` 和 `//` 是**居中中缀**（infix）：

```
甲  %  乙
↑      ↑
左     右
```

视觉配对无需前瞻。

## 五、认知自然性测试

### 5.1 命令式汉语 vs 嵌入式介词结构

周道中已有的结构：

| 句式 | 形式 | 结构 |
|------|------|------|
| 绑定 | `设甲为乙。` | 动词 + 主语 + 补语 |
| 条件 | `如果甲，就乙。` | 连词 + 分句 |
| 遍历 | `从甲中，每取一项记作乙，就丙。` | 介词短语 + 动宾 |
| 函数 | `以甲为所得。` | 介词 + 宾语 + 补语 |

这些全是**句子级结构**——句号结束一个完整语句。`对……取余` 和 `对……取整` 是**表达式内结构**——它们出现在 `设甲为……` 或 `如果……就` 的内部。

周道之前没有在**表达式层**使用过这种「介词包裹操作数」的结构。`设甲为乙` 的 `为` 是连词，`甲` 和 `乙` 都是简单操作数。但 `对`……`取余` 需要在中间嵌套任意复杂表达式。

### 5.2 直觉测试

读下面几行，感受阅读节奏：

```
设总页数 = 总条数 // 每页条数 + 1。
设桶号 = 用户序号 % 桶数。
设当前日期 = UTC天数 + 时区偏移 // 24。
如果行号 % 26 = 0 且行号 != 0，就插入分页符。
设回绕索引 = (当前位置 + 总长度 - 步长) % 总长度。
设实际索引 = (当前位置 + 偏移) % 总长度。
```

```
设总页数 = 总条数对每页条数取整 + 1。
设桶号 = 用户序号对桶数取余。
设当前日期 = UTC天数 + 时区偏移对24取整。
如果行号对26取余 = 0 且行号 != 0，就插入分页符。
设回绕索引 = (当前位置 + 总长度 - 步长) 对总长度取余。
设实际索引 = (当前位置 + 偏移) 对总长度取余。
```

**观察记录：** 第二组的 `对` 在长表达式尾端出现时，第一次扫读会先读到 `(当前位置 + 总长度 - 步长)`，然后看到 `对`——这时需要回头看 `对` 前面到底是什么。`%` 和 `//` 不需要这种回溯。

## 六、预裁决

### 6.1 `%` — 强倾向于保留符号

| 因素 | 倾向 |
|------|------|
| 正整数场景汉语优于符号 | ✅ 成立 |
| 全局语义（负数/浮点）汉语与 Python 有偏差 | ⚠️「余」误导 |
| 介词结构组合性弱于符号 | ⚠️ 有证据 |
| 最佳场景（奇偶/分组）与符号差距 | 不致命，符号一样清楚 |
| 需要「对…取余」替换的频率 | 中等（日常代码中 % 出现频率不低） |

**当前建议：** 保留 `%` 为符号源码，LSP 工具说明解释为"按 Python 规则取余"。不排除在正整数上下文提供 `对…取余` 作为兼容输入，但规范形式为 `%`。

### 6.2 `//` — 强倾向于保留符号

| 因素 | 倾向 |
|------|------|
| 汉语无准确对应（「整除」≠ floor） | ✅ 成立 |
| 「对…取整」组合性弱 | ⚠️ 有证据 |
| `//` 在程序员群体中有广泛认知 | ✅ 「整除」口头上已普及 |
| 分页/索引场景出现频率高 | ✅ 高频 |
| 负数场景有实际用途 | ✅ 时区等 |

**当前建议：** 保留 `//` 为规范符号。LSP 解释为"向下取整除法"。

### 6.3 收回「对……取余」候选

这条审计最大收获是发现**介词结构不适合作为周道表达式层的中缀运算符**。周道的汉语优势在**句子层**（`如果……就……`、`从……中每取一项记作……就……`、`以……为所得`），这些句子的操作数大多是简单名称。一旦操作数本身是复合表达式，介词包裹结构就开始吃力。

因此：

```
候选撤回：甲对乙取余
候选撤回：甲对乙取整
保留符号：甲 % 乙
保留符号：甲 // 乙
```

此结论受补充原则支持：

> 翻译会变成长句或技术黑话时，宁可保留符号并由工具解释。
