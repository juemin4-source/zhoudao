# 表达式层裁决 v0.10.5

> 依据「周道翻译程序关系，不朗读数学符号。程序动作使用汉语，
> 数学表达式使用键盘原生符号」原则作出的逐项裁决。
> 本文件作为 `ZHOUDAO-LANGUAGE-SPEC-1.0.md` 表达式章节的临时修订依据，
> 待正式合并。

---

## 一、总则

**周道形式化的是中国程序员脑内的内语，不是翻译 Python 文本。**

内语里已经是符号本身的 → 保留符号
内语里依赖汉语结构重建的 → 写成汉语
汉语表达选最自然的口语形式，不逐字对应英语语法

---

## 二、规范源码字符集（表达式区域）

表达式区域只使用以下 ASCII 符号，Unicode 数学字形仅用于显示层：

```
  +   加法 / 一元正号
  -   减法 / 一元负号
  *   乘法
  /   除法
  **  幂
  =   相等判断
  !=  不等判断
  >   大于
  <   小于
  >=  大于等于
  <=  小于等于
```

### 格式规则

二元运算符两侧各加一个半角空格：

```
设判别式为乙 * 乙 - 4 * 甲 * 丙。
如果判别式 >= 0，就……
```

一元负号不加空格：

```
设温度为 -3。
设结果为 -(甲 + 乙)。
```

## 四、逐运算符裁定

### ✅ 通过：纯数学符号（保留键盘原生形式）

| 语义 | 规范形式 | 理由 |
|------|---------|------|
| 加法 | `甲 + 乙` | 小学建立直觉，内语直读符号 |
| 减法 | `甲 - 乙` | 同上 |
| 乘法 | `甲 * 乙` | `*` 键盘通用 |
| 除法 | `甲 / 乙` | `/` 键盘通用 |
| 幂 | `甲 ** 乙` | Python 已有直觉，键盘直输；拒绝 `^`（XOR 歧义） |
| 相等 | `甲 = 乙` | 数学第一意义就是相等；周道无赋值冲突 |
| 不等 | `甲 != 乙` | 兼容 Python/键盘 |
| 大于 | `甲 > 乙` | 小学建立直觉 |
| 小于 | `甲 < 乙` | 同上 |
| 大于等于 | `甲 >= 乙` | 键盘通用 |
| 小于等于 | `甲 <= 乙` | 键盘通用 |

### ✅ 通过：程序关系（保留汉语）

| 语义 | 规范形式 | 理由 |
|------|---------|------|
| 逻辑且 | `甲且乙` | `∧` 无普遍直觉 |
| 逻辑或 | `甲或乙` | `∨` 无普遍直觉 |
| 逻辑非 | `并非（甲）` | `¬` 无普遍直觉 |
| 成员 | `甲在乙中` | 程序关系，汉语显化 |
| 非成员 | `甲不在乙中` | 同上 |
| 身份 | `甲就是乙` | 同上 |
| 非身份 | `甲不是乙本身` | 同上 |
| 字段访问 | `甲的乙` | `的` 是中文天然属格 |
| 下标 | `甲［乙］` | 不涉及数学符号 |

### ✅ 通过：状态变化动作（保留汉语动词）

```
使数量加1。
使数量减1。
使数量乘2。
使数量除2。
使状态变为【完成】。
使自己的余额减支出。
```

**规则：** `使` 后必须跟汉语变化动作，禁止 `使 X + Y` 形式。
**诊断：** `"使"后需要变化动作。若要改变数量，请写"使数量加1。"若只想计算，请写"数量 + 1。"`

### ✅ 已裁决：`//` — 保留符号

**状态：** APPROVED（保留 `//`）
**语义：** 向下取整除法。先求商，再向负无穷取整。
**示例：**
```
设页号 = 项目序号 // 每页数量。
设中位 = （左 + 右）// 2。
设本地日期 = UTC天数 + 时区偏移 // 24。
```

**所有汉语候选已拒绝（标记为 REJECTED_SURFACE_PROPOSAL，非弃用语法）：**
```
甲整除乙               ← REJECTED（「整除」≠ floor）
甲对乙取整             ← REJECTED（介词结构组合性弱）
甲除以乙后向下取整     ← REJECTED（不适合作为中缀运算符）
```

**工具悬停说明：**
> 向下取整除法。先求商，再向负无穷取整。
> `5 // 2` → `2`　　`-5 // 2` → `-3`

### ✅ 已裁决：`%` — 保留符号

**状态：** APPROVED（保留 `%`）
**语义：** Python 模运算（完整继承，含负数/浮点规则）。
**示例：**
```
设桶号 = 用户序号 % 桶数。
如果数量 % 2 = 0，就显示【偶数】。
设回绕索引 = （当前位置 + 偏移）% 总长度。
```

**所有汉语候选已拒绝（标记为 REJECTED_SURFACE_PROPOSAL，非弃用语法）：**
```
甲对乙取余     ← REJECTED（介词结构组合性弱，负数时「余」误导）
甲余乙         ← REJECTED（过于压缩，负数语义不清）
甲模乙         ← REJECTED（技术黑话，偏离内语）
```

**工具悬停说明：**
> 余数运算。结果符号遵循除数，满足：
> `被除数 = 除数 × 向下取整商 + 余数`
> `5 % 2` → `1`　　`-5 % 2` → `1`　　`5 % -2` → `-1`

### 🔮 明确不支持（表达式区域）

| 结构 | 理由 | 预期版本 |
|------|------|---------|
| 链式比较 `a < b < c` | 语法复杂性 | 013+ |
| `*args` / `**kwargs` | 非基础 | 012+ |
| 匿名函数 `lambda` | 非基础 | 013+ |
| 推导式 | 非基础 | 013+ |
| 运算符重载 | 需完整类模型 | 014+ |

## 六、表达式层三段划分

所有表达式关系按认知类型分为三层：

| 层 | 类别 | 处理 | 示例 |
|----|------|------|------|
| **数学与数量关系** | `+ - * / ** // % = != < <= > >=` | **符号** | `甲 + 乙`、`甲 % 乙` |
| **程序语义关系** | 成员、身份、字段 | **汉语** | `甲在乙中`、`甲就是乙`、`甲的乙` |
| **逻辑命题关系** | 且、或、并非 | **汉语** | `甲且乙`、`并非（甲）` |

**划分依据：** 左栏的符号已经通过数学教育建立了跨语言直接认知；中栏和右栏表达的是程序特有的关系，依赖汉语结构重建。

**注意：** `在……中` 虽然是关系表达式，但它表达的是程序中的成员关系，通常作为一个完整命题出现；而 `%`、`//` 则会进入高密度算术嵌套。两者承受的组合压力不同，不应按「是否符号」一刀切。

## 七、表达式设计门禁

任何未来的汉语运算表达式候选，必须同时通过以下测试：

| 测试项 | 说明 |
|------|------|
| 简单操作数 | `甲 候选 乙` |
| 复合左操作数 | `（甲 + 乙）候选 丙` |
| 复合右操作数 | `甲候选（乙 + 丙）` |
| 同类嵌套 | `甲候选乙候选丙` |
| 混合优先级 | `甲 + 乙候选丙` |
| 条件中使用 | `如果甲候选乙 = 0` |
| 函数参数中使用 | `处理（甲候选乙）` |
| 连续扫读 | 放在 5 行真实代码中，能否一眼看懂 |

这条门禁来自 `%` 和 `//` 审计的教训：汉语介词结构 `对……取` 在简单操作数时好看，一旦进入复合表达式，组合性弱于符号。

> **周道翻译程序，不朗读公式。**

## 九、迁移说明

### 兼容输入（临时，仅 migration 期）

```
==  作为 = 的兼容输入，Formatter 统一为 =
>=  作为 >= 的同义输入（当前实现不变项，注明即可）
<=  作为 <= 的同义输入（同上）
```

**不设三级并存：** `等于`、`==`、`=` 不是三个平级标准。规范只有 `=`。

### 编译器内部命名

```
加法节点:   Add          (不变)
乘法节点:   Mul          (不变)
幂节点:     Pow          (不变)
负号节点:   Neg          (不变)
等号节点:   Eq           (不变，发射输出 =)
不等节点:   NotEq        (发射输出 !=)
取余节点:   Mod          (内部名不变，表层发射 "对...取余")
整除节点:   FloorDiv     (内部名不变，表层待定)
```
