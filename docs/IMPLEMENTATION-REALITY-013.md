## 主要断链

### 断链 1：`从周道源文件...中引入` 名称未注入 （WP2）

**现状**：
- Parser 正确解析 `从周道源文件《helper》中引入双倍。`
- Lowering 正确生成 `从本地模块引入IR`（模块名="helper", 名称=["双倍"]）
- `ModuleLoader.load_module()` 只处理 `本地模块引入IR`（整体引入），不处理 `从本地模块引入IR`
- 依赖模块被加载到缓存，但选择引入的名称**从未进入当前模块命名空间**
- `exec(code, 模块环境)` 执行时，`双倍` 不在环境中 → `NameError: name '双倍' is not defined`

**影响**: 所有使用 `从周道源文件...中引入` 的多文件项目都无法正确运行。

### 断链 2：LSP 吞掉词法和语法错误 （WP16）

**现状**：
- 无效字符 `@` → LSP 返回 0 条诊断
- 未定义名称 → LSP 返回 1 条诊断（语义分析阶段可工作）
- 词法阶段错误在 LSP 中被静默吞掉

**影响**: 编辑器内无法看到词法/语法错误标记。

### 断链 3：RuntimeTraceback 不识别多模块帧 （WP10）

**现状**：
- `提取周道帧` 只检查 `f_code.co_filename == '<周道>'`
- 模块编译时使用 `.zd` 文件路径作为 filename
- 模块执行栈帧的 filename 是 `.zd` 路径，不是 `<周道>`
- 导致模块执行错误无法映射回周道源码

**影响**: 所有模块执行错误都丢失源码位置。

### 断链 4：BackendSourceMap 未接入后端正轨 （WP9）

**现状**：
- `backend_source_map.py` 文件存在，定义了 SourceMap 和节点跟踪机制
- `PythonAstBackend` 未使用该组件
- 后端使用内部的简易 `行映射: dict[int, 源码位置]` 进行单文件映射

**影响**: 跨模块节点溯源无统一入口。

### 断链 5：CrossModuleMap 未接入运行时 （WP10）

**现状**：
- `cross_module_map.py` 存在
- `ModuleLoader` 的 `_跨模块映射` 字典存储了路径 → 行映射
- 但 `运行时错误` 和 `RuntimeTraceback` 未消费此映射

**影响**: 跨模块运行时错误只能映射最后执行的文件，不能追溯原始周道源码链。

### 断链 6：整体引入 `引入周道源文件《X》` 的绑定语义未定义

**现状**：
- 该语法可解析和降低
- `ModuleLoader` 加载了依赖模块
- 但被引入模块的公开名称以何种形式注入当前命名空间——现行规范未明确
- 当前实现执行了依赖模块但未绑定名称到当前域

**影响**: 需要用户裁决 `引入周道源文件《X》` 的绑定语义。
