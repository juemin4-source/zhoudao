## 四项阻塞级修复

### R1. 精确名称宪法恢复

**问题：**
- `{用户的姓名}` 被词法后处理拆成成员访问
- `{错误内容}` 被 Parser 当成上下文名称拦截
- `{甲须大于0}` 等边界被错误合并

**宪法原文（IDENTIFIER-SPEC.md）：**
- 精确名称内部永不重新分词
- 精确名称永不被上下文劫持

**修复：**
1. Lexer 中精确名称（`是否精确=True`）跳过所有后处理
2. Parser 中 `名称来源 == "EXACT"` 跳过上下文关键字判断（已有的 `if 是精确: return 变量(...名称来源="EXACT")` 但语义分析器未遵守）
3. Semantic Analyzer 中精确名称直接进入名称解析，不触发上下文判断

**验证：**
```
设{用户的姓名}为【张三】。  → 必须绑定精确名称"用户的姓名"
显示{用户的姓名}。           → 必须读取同一绑定
设{错误内容}为【测试】。     → 必须在异常上下文外作为普通名称使用
```

### R2. Core IR 构造契约恢复

**问题：**
- `函数定义IR("fn_c", [], [报错IR(...)])` 在新字段 `参数默认值` 插入后把体当成默认值
- 多层回溯测试出现 `empty body on FunctionDef`

**修复：**
1. 所有 IR 节点的可变字段必须追加到最后，不改变已有位置参数顺序
2. 或提供兼容构造器
3. 检查所有历史 `函数定义IR(...)` 和 `类别方法IR(...)` 构造调用

### R3. 词法确定性恢复

**问题：**
- `_拆分嵌入操作符` 后处理 + Parser 中的 `名称.endswith("为")` 等正在滑向"先扫大字符串再猜"
- `甲须`、`用户的姓名`、`数量减`、`错误内容` 等边界被错误合并或拆分

**修复：**
1. 移除 `_拆分嵌入操作符` 后处理
2. 恢复 Lexer 的确定性分词（关键字边界打断标识符，但不打断复合词中的操作符子串）
3. 移除 Parser 中所有 `name.endswith("为")`、`val.startswith("错误类型是")` 等补丁
4. 验证所有 003 测试通过

### R4. 生成器文章结构闭环

**问题：**
- 生成器程序生成的 Python 把 `运行如下` 塞进了函数体
- 验收报告中的"普通生成器通过"是假的

**修复：**
1. 修复 Parser 中定义体收集的终止条件（`K_ENTRY` 等顶层关键词必须终止体）
2. 验收用真实输出验证
