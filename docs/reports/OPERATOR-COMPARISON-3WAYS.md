# 运算符三版对照

> 目的：在同一组真实代码上并排比较全汉语、ASCII数学符号、Unicode数学字形三种写法。
> 验证「程序结构用汉语显化，数学结构保留数学」这条原则是否站得住。

---

## 测试素材 1：二次方程判别式

| 版本 | 源码 |
|------|------|
| **A 全汉语** | `设判别式为乙乘乙减4乘甲乘丙。`<br>`如果判别式不少于0，就……` |
| **B ASCII 符号** | `设判别式为乙 * 乙 - 4 * 甲 * 丙。`<br>`如果判别式 >= 0，就……` |
| **C Unicode 数学** | `设判别式为乙 × 乙 − 4 × 甲 × 丙。`<br>`如果判别式 ≥ 0，就……` |

### 扫读对比

A 行需要**先分词再计算**：「乙乘乙减4乘甲乘丙」——是 `(乙*乙) - (4*甲*丙)` 还是 `乙* (乙-4) *甲*丙`？线性汉语抹掉了运算结构。

B 和 C 保留了视觉骨架，`*` 和 `×` 的视觉权重不同——`×` 更醒目，但 `*` 键盘直接输入。

## 测试素材 2：绝对值函数

| 版本 | 源码 |
|------|------|
| **A 全汉语** | `设绝对值（数），如果数不少于0，就以数为所得，不然就以0减数为所得。` |
| **B ASCII 符号** | `设绝对值（数），如果数 >= 0，就以数为所得，不然就以 0 - 数为所得。` |
| **C Unicode 数学** | `设绝对值（数），如果数 ≥ 0，就以数为所得，不然就以 0 − 数为所得。` |

### 扫读对比

三者在「设绝对值（数）」和「以数为所得」上完全一致。差异只存在于条件判断和表达式部分。

A：`0减数` — 动词的「减」在纯表达式里读起来像在描述动作，但这里只是算术。
B：`0 - 数` — 一目了然。
C：`0 − 数` — 和 B 几乎等价，减号字形更专业。

## 测试素材 3：信号灯（动作 vs 表达式分离）

| 版本 | 源码 |
|------|------|
| **A 全汉语** | `定义信号灯类别的切换（）如下：`<br>  `如果自己的状态等于【红】，就使自己的状态变为【绿】；`<br>  `不然如果自己的状态等于【绿】，就使自己的状态变为【黄】；`<br>  `不然使自己的状态变为【红】。` |
| **B ASCII 符号** | `定义信号灯类别的切换（）如下：`<br>  `如果自己的状态 = 【红】，就使自己的状态变为【绿】；`<br>  `不然如果自己的状态 = 【绿】，就使自己的状态变为【黄】；`<br>  `不然使自己的状态变为【红】。` |
| **C Unicode 数学** | `定义信号灯类别的切换（）如下：`<br>  `如果自己的状态 = 【红】，就使自己的状态变为【绿】；`<br>  `不然如果自己的状态 = 【绿】，就使自己的状态变为【黄】；`<br>  `不然使自己的状态变为【红】。` |

### 扫读对比

这里最关键的变化是 `等于` → `=`。A 的 `等于` 读起来和汉语「等于」一样自然，没有认知负担。但对比后：

- A 的 `等于` 用了 2 字 + 2 空格宽度，B/C 的 `=` 用 1 字符。
- 差异虽小，但条件判断是高频结构，累积下来视觉密度差异明显。
- 如果判断条件是 `年龄 ≥ 18`，视觉上比 `年龄 >= 18` 更干净，且 `>=` 始终是两字符宽。

B 和 C 在此例中完全等价——这里不涉及乘除号选择。

**但「使自己的状态变为【绿】」这个动作没有变更**——你明确说过 `使` + 动词属于动作，不走符号化。

## 测试素材 4：生成器流水线（长表达式）

| 版本 | 源码 |
|------|------|
| **A 全汉语** | `定义加倍（来源）如下：`<br>  `从来源中，每取一项记作数，就依次给出数乘2。`<br><br>`定义范围（上限）如下：`<br>  `设i为0。`<br>  `当i小于上限时，一直依次给出i，并使i加1。` |
| **B ASCII 符号** | `定义加倍（来源）如下：`<br>  `从来源中，每取一项记作数，就依次给出数 * 2。`<br><br>`定义范围（上限）如下：`<br>  `设i为0。`<br>  `当i < 上限时，一直依次给出i，并使i + 1。` |
| **C Unicode 数学** | `定义加倍（来源）如下：`<br>  `从来源中，每取一项记作数，就依次给出数 × 2。`<br><br>`定义范围（上限）如下：`<br>  `设i为0。`<br>  `当i < 上限时，一直依次给出i，并使i + 1。` |

### 扫读对比

这里出现了一个边缘情况：**`使i加1`**。

按新原则，`使` 是状态修改**动作**，所以 `加1` 是动作的组成部分，不是纯数学表达式。但如果写成 `并使 i + 1`，`使` 后面跟的是一个表达式，语义上变成了：

> 使 i+1（让 i+1 这个值发生什么？）

而不是：

> 使 i 加 1（让 i 增加 1）

所以「动作内的算术」和「纯表达式中的算术」可能需要不同处理：

```
使数量加 1。        ← 动作：增长
设结果为数量 + 1。   ← 表达式：计算
```

这个区分很微妙，但在阅读中自然成立——`使` 预告了动作，`设` 预告了绑定。

## 测试素材 5：过滤偶数的条件表达式

| 版本 | 源码 |
|------|------|
| **A 全汉语** | `从来源中，每取一项记作数，就（如果数余2等于0，就依次给出数）。` |
| **B ASCII 符号** | `从来源中，每取一项记作数，就（如果数 % 2 = 0，就依次给出数）。` |
| **C Unicode 数学** | `从来源中，每取一项记作数，就（如果数 % 2 = 0，就依次给出数）。` |

### 扫读对比

`数余2等于0` 在 A 版中一口气 6 个汉字，分词后是「数 余 2 等于 0」。「余」字在这里是**程序运算**（取模），不是小学数学概念。`%` 对非程序员是百分号，对程序员是取模——汉语 `余` 对非程序员更友好。

这里你的原则正好适用：`%` 没有跨语言直接认知，所以保留汉语。

B/C 的 `数 % 2 = 0` 更短，但 `%` 的认知成本不低。这条目初步判断：**汉语更优**。

## 测试素材 6：多步骤计算（含减法结合括号）

| 版本 | 源码 |
|------|------|
| **A 全汉语** | `设结果为（5加3）乘（5减2）。`<br>`设温差为最高温减最低温。` |
| **B ASCII 符号** | `设结果为（5 + 3）*（5 - 2）。`<br>`设温差为最高温 - 最低温。` |
| **C Unicode 数学** | `设结果为（5 + 3）×（5 − 2）。`<br>`设温差为最高温 − 最低温。` |

### 扫读对比

A：`（5加3）乘（5减2）`——括号内的 `加` 和 `减` 是汉语动词，但 `乘` 连接两个括号结果。这里汉语和运算混在一起，视觉上 `（5加3）` 需要先转换成 `(5+3)` 才能计算。

B：`（5 + 3）*（5 - 2）`——直接计算，零转换。注意括号用的是全角 `（）` 而非半角 `()`，在全角符号体系中视觉统一。

C：`（5 + 3）×（5 − 2）`——`×` 比 `*` 更宽，视觉上更对称，但输入不便。

温差行也值得注意：`最高温 - 最低温` 或 `最高温 − 最低温` 都比 `最高温减最低温` 更接近数学公式的自然写法。

## 需要你判定的分界点

### 1. `使` + 算术动作

```
使数量加 1。     ← 动作中的汉语
使数量 + 1。     ← 混搭，可能非法
数量 + 1         ← 纯表达式，符号
```

**判定点**：`使 X 动词 值` 是否强制要求动词形式，禁止 `使 X + 值`？

### 2. 等号 `=` 的赋值混淆风险

Python 程序员初次见到：

```
如果年龄 = 18，就……
```

会反射性认为"这是赋值不是比较"。但这是**一次学习永久消除**的误解——看到第一个 `设X为Y` 后就知道 `=` 是相等。

**判定点**：是否接受 `=` 作为比较运算符，还是保留 `=` 给未来其他用途？

### 3. 整除和取余的符号化候选

如果不翻译数学，那 `//` 和 `%` 也是数学符号（虽然普及度低）：

```
甲 // 乙         ← 对程序员是整除，对非程序员不是
甲 % 乙          ← 对程序员是取模，对非程序员是百分号
```

你的原则是"已有跨语言直接认知的符号保留"，`//` 和 `%` 显然不满足条件。

**判定点**：确认 `整除` 和 `余` 保留汉语。

### 4. 字形层的正交性

键盘输入 `× ÷ ≥ ≤ ≠` 需要：

- Windows: `win+.` 符号面板 / 输入法特殊符号
- Mac: `option+=` 等
- 编辑器 snippet / 自动补全

是否把**输入法层**和**源码规范层**分开？比如：

- 规范接受 `>= <= !=` 作为源码
- 格式化器可以自动渲染/显示为 `≥ ≤ ≠`
- LSP 可以提示"要不要显示为数学符号"

这样键盘友好和数学显示不冲突。

### 5. `^` 的优先级

如果 `^` = 幂，它需要比乘除高，接近 Python 的 `**` 优先级：

```
2 * 3 ^ 4  →  2 * (3^4) = 162
```

但 C 系语言中 `^` 是 XOR，优先级低于加减。这可能导致 Python 背景用户的直觉冲突。

**判定点**：`^` 的优先级定在哪里？
